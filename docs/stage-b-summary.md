# 阶段 B 总结

阶段 B 的目标是把 Coach 从“已经装配好的身份”推进到“开始具备连续训练价值的文本闭环”。这一阶段没有去碰新的 graph，也没有继续扩展飞书协议层，而是把注意力集中在赛前建议、赛后复盘和结构化沉淀三件事上。这样做的原因很直接：如果没有稳定的领域规则和可回写的数据层，前面的 Agent 装配再完整，也只能给出一次性的泛化回答，很难形成真正会越来越懂用户的教练体验。

这一阶段首先完成了 `prematch` 文本路径。实现上没有把逻辑塞进一个大 prompt，而是新增了规则层，让系统可以按优先级读取 `coach_profile.json`、最近 review logs、`memory.json` 和天气上下文，再组装成训练重点、热身建议和风险提醒。这样带来的好处是，个性化依据是可测试、可解释的：如果命中了结构化弱项，就应该引用结构化弱项；如果没有历史记录，就应该明确回退到通用建议。B1 的单测正是围绕这个行为边界来写的。

接着完成的是 `postmatch` 文本路径。这里的关键不是“生成一段看起来像复盘的话”，而是把自然语言拆成可继续利用的结构化观察。当前实现已经能从文本里区分技术问题、进步点、下次重点和情绪性描述，并避免把“有点着急”“没什么明显技术问题”这类内容误判为技术事实。这个约束非常重要，因为一旦情绪表达被误写成长期弱项，后面的赛前建议就会被错误历史污染。

在 B3 里，阶段 B 才真正形成闭环。`postmatch` 的结果不再只停留在内存里，而是会写回 `coach_profile.json` 和 `memory/reviews/YYYY-MM-DD.md`。前者负责保存稳定结构，如技术弱项、近期关注点、进步项和最近几次复盘摘要；后者负责按日期沉淀原始训练事件，回答“最近到底发生了什么”。这两个层次的分工是这一阶段最重要的工程决策之一：结构化档案适合做稳定状态读取，日期日志适合做近期事实追溯，两者组合起来，比只依赖一个通用 `memory.json` 更适合教练场景。

从测试结果看，阶段 B 的核心能力已经具备最小可回归基础。`test_coach_prematch_rules.py` 覆盖了赛前建议的个性化和降级逻辑，`test_coach_postmatch_extraction.py` 覆盖了赛后抽取的结构化边界，`test_coach_profile.py` 和 `test_coach_integration_flow.py` 则验证了“赛后写回 -> 下次赛前读取”的最小链路。这意味着阶段 B 的价值不只是新增了几个模块，而是把“上一次训练结果会影响下一次建议”这件事从 spec 写成了代码事实。

当前阶段的残留风险也很明确。第一，文本主闭环目前还是规则层优先，尚未进入更细的模型编排和置信度评估，因此复杂混合输入的泛化能力还有限。第二，`coach_profile.json` 的合并策略目前足够支撑单用户 MVP，但还没有扩展到更复杂的趋势分析和历史衰减策略。第三，天气上下文和更产品化的飞书输出还没有并入阶段 B 主链路，这部分会留给阶段 C 来完善。

阶段 C 的进入条件已经具备：结构化档案可读、日期日志可追、赛前赛后文本路径可测。接下来的重点不再是“有没有主闭环”，而是“上下文增强是否真的能让建议质量更稳”，也就是 Weather MCP、飞书输出结构和 structured logs 的接入效果。
