# 阶段 C 总结：上下文增强
阶段 C 的目标不是继续扩功能数量，而是把 Coach 从“能回答”推进到“回答更像一个可用产品”。这一阶段完成了三个关键动作：第一，把天气上下文接进赛前建议链路；第二，把飞书回复从单块文本改成结构稳定的卡片；第三，把主链路运行数据沉淀成 structured logs。这样做的意义在于，系统现在不仅能给建议，还能根据外部条件调整建议、用更清晰的形式展示建议，并且把后续排障和评估需要的数据记录下来。

`C1` 的重点是天气上下文。此前赛前建议虽然已经支持传入 `weather` 字典，但缺少一层稳定的适配和降级逻辑，实际工程里无法认为已经“接入 Weather MCP”。这次补上了天气适配器，支持把 MCP 风格的嵌套 payload 规整成统一结构，并在工具缺失、调用失败或返回内容不完整时输出降级上下文，而不是让主流程报错。赛前建议层也从“简单拼接天气提醒”升级成真正使用天气信号去调整补水、单次训练时长和起始强度。高温与普通天气现在会产生明显不同的输出，满足了这一阶段的核心目标。

`C2` 解决的是产品表现层问题。飞书原本的 interactive card 只是一个 markdown 容器，虽然能显示内容，但结构感弱，实际使用时更像把聊天文本包了一层卡片壳。现在新的卡片构建逻辑会尝试把 Coach 回答拆成“建议摘要、重点项、热身建议、风险提示、下次建议”等固定区域，并加上 header。这样一来，用户在手机端看回复时能更快抓住重点，也为后续继续做更强的产品化展示留出了稳定的模板入口。更重要的是，这次改动没有破坏既有 running card patch 机制，说明展示层增强没有影响消息主链路稳定性。

`C3` 补的是工程可观测性。当前阶段还不适合一上来就接 dashboard，但如果连基础日志都没有，后续做评估和性能优化就会缺少抓手。因此这次在 `ChannelManager` 收口处补了结构化运行日志，统一记录 `assistant_id / agent_name` 路由信息、请求耗时、response 长度、artifact 数量、token usage，以及 memory 命中情况。memory 命中目前采用“显式字段优先、引用上下文回退”的 best-effort 方式，已经足够支持固定样本回放和日志校验，也为后续 D2、D3 的评估与统计打好了底。

从交付结果看，阶段 C 已经把“上下文增强”这个主题闭环完成：天气属于外部上下文，飞书卡片属于展示上下文，structured logs 属于运行上下文。三者叠加后，Coach 的下一阶段工作就不再是补基本工程能力，而是进入更高价值的增强路径。阶段 D 的进入条件现在已经成立，包括：主链路可回归、输出结构基本稳定、运行指标可记录、外部上下文接入已经有样板实现。下一步可以进入 Web 图片分析、离线评测和性能/成本统计这些更偏产品质量和评估体系的工作。
