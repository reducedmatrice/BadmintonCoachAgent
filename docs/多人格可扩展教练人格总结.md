# 多人格可扩展教练人格总结

## 背景

之前 `badminton-coach` 的人格能力主要依赖 `SOUL.md` 和零散的人格资产文件。这样做在只有一个默认人格时问题不大，但一旦后面要支持更多人格，甚至让飞书用户自行切换，就会出现两个明显问题：第一，教练底座和具体人格耦合在一起，新增人格时容易继续改动主链路；第二，仓库里虽然有一些 persona 资产，但并没有形成统一、稳定、可切换的运行时入口，长期会越来越乱。

这次改造的目标不是单纯把郭德纲风格接进去，而是顺手把“人格”从教练底座里拆出来，做成后续可以扩展的能力。也就是说，现在要让郭德纲人格真实生效，但更重要的是，以后再加别的人格时，不需要再重做一次架构。

## 实现方式

这次实现把人格分成两层。第一层是 `SOUL.md`，它只保留稳定不变的教练身份、职责边界和安全规则，比如羽毛球场景、风险优先、不能做医疗诊断、输出尽量自然段。第二层是可切换人格，统一收敛到 `personalities/<personality_id>/` 目录下，每个人格都有自己的 `persona.md` 和 `meta.json`。其中 `persona.md` 负责自然语言风格描述，`meta.json` 负责人格标签和结构化 style 配置。

在运行时，agent 会先读取教练底座，再根据 `default_personality` 或显式传入的 `personality_id` 选择具体人格，把选中的 persona 注入 prompt。同时，程序化出口也不再写死郭德纲，而是从人格的结构化 style 配置里拿默认语气、追问方式和表达倾向。这样一来，prompt 层和代码层都能对齐同一个人格来源。

这次默认接入的是 `guodegang` 人格。它保留了嘴贫、短句、会翻旧账、会用比喻提醒用户的表达特点，但仍然受教练底座约束，在疼痛、恢复和风险场景下会主动收口，不会为了“像郭德纲”牺牲安全性和判断质量。

## 价值
这次改造最大的价值，不是多了一个郭德纲风格，而是把人格正式变成了一个可管理、可配置、可扩展的系统能力。后续如果要增加更多人格，只需要新增人格目录和配置，不必继续改教练主逻辑。未来如果要开放给飞书用户切换，也已经有了明确的运行时入口，只需要把选中的 `personality_id` 透传进来。

从工程角度看，这次也顺手厘清了边界：`SOUL` 负责稳定底座，`persona` 负责可切换风格，结构化 style 只影响表达，不覆盖 route、risk 和 safety。这样既能让 agent 更有“人味”，又能避免人格系统反过来污染核心业务判断。对于后续继续做多人格、个性化和产品化，这一步是必要的基础设施。
