# 阶段 D 总结：Health 图片链路与评估体系
阶段 D 的核心目标，是把 BadmintonCoachAgent 从“主要靠文本输入”的教练，推进到“开始具备多模态理解能力和可量化评估手段”的阶段。相比前面的 A、B、C，阶段 D 更像一次能力补强：不是重新定义主链路，而是让已有链路开始支持更接近真实使用场景的输入形式，并给后续优化建立衡量基线。

这一阶段的第一项工作是 `D1`，也就是 Web 图片分析链路。这里没有为了追求“多模态”而直接接一个复杂的 OCR 或视觉编排系统，而是根据当前仓库已具备的事实能力做了收敛。Web Workspace 本身已经有上传能力，agent 侧也已经有 `uploaded_files` 和 `view_image` 相关中间件，因此阶段 D 重点不是重复实现上传，而是把 Health 场景下的常见截图输入整理成结构化观察和保守恢复建议。当前实现已经支持三类最常见样本：心率总结截图、睡眠恢复截图、训练负荷/恢复时间截图。这样做的价值在于，即使还没有进入更复杂的 OCR/STT 体系，Coach 也已经能围绕“截图里看到了什么”给出可执行建议，而不是退回纯聊天。

`D1` 里另一个关键点，是明确了图片工作流边界。健康类图片输入不会被当作“万能可解释图像”，而是按保守策略处理：优先提取可见指标，无法确认的数据就标为 unknown，不去臆造隐藏字段，也不把运动数据包装成医学诊断。这个边界非常重要，因为一旦在健康场景里过度自信，模型输出的风险就会迅速上升。当前的 `health` skill 也同步补上了图片场景工作流，要求先检查 `<uploaded_files>`，再使用 `view_image`，并把截图类型收敛到少数几类稳定样本上。

阶段 D 的第二项工作是 `D2`，也就是离线评测体系。到这一步，项目已经具备赛前、赛后、健康、天气、日志、图片这些能力，如果没有离线样本集和统一评分结构，后续每次改动都会很难判断是在变好还是变差。因此这一阶段补上了离线评测模块、样本文件和脚本入口。当前样本集覆盖了 prematch、postmatch 和 health image 三类场景，评测维度则收敛到 route、structure、actionability、grounding、safety 五个方面。这里没有直接接真实 LLM judge，而是先把评分结构和 markdown 报告固定下来，为后续替换成更复杂的 judge prompt 留接口。

`D3` 则把阶段 C 打下的 structured logs 正式利用起来，做成可汇总的指标输出。现在系统不仅会在请求完成时记录 route、latency、token usage 和 memory hits，还能通过独立脚本对日志文件做聚合，输出整体请求数、错误率、P50/P95 latency、平均 token，以及按 route 细分的统计结果。这个能力的意义不只是“看日志更方便”，而是给后续做性能优化、成本比较和场景稳定性分析提供了统一入口。这样一来，阶段 D 就把“内容评估”和“运行评估”两条线都补齐了。

从结果上看，阶段 D 已经把项目从“能跑 demo”推进到了“开始具备评估和增强基础设施”的状态。Web 图片链路让 Health 场景不再只依赖用户手打文本，离线评测让建议质量有了固定样本可回放，日志汇总则让延迟和 token 统计变成可批量分析的指标。阶段 E 的进入条件因此也已经具备：一方面，多模态已经有 Web 端路径作为参考样板；另一方面，项目已经有足够的结构化日志和测试保护，可以支撑进一步把飞书图片/语音/文件入站与主动提醒接入主流程。
