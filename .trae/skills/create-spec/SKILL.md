# Create Spec

基于用户输入产出三份文档：
- `dev_spec.md`
- `tasks.md`
- `checklist.md`

必须执行 Human-in-the-Loop：先收集必需信息，再分阶段与用户确认，确认后再进入下一步。
`dev_spec.md` 的结构必须严格遵循 `reference/spec-format.md`，不得自行改格式。

## 核心原则

1. 先确认项目目标与核心特性，再做架构。
2. 不确认就不落盘关键决策。
3. 不确定实现用占位，不编造细节。
4. 每个核心特性必须映射到任务与验收。
5. 输出必须工程可执行、可验证、可追踪。

## Pipeline

按顺序执行，不得跳步。

### Phase 1：输入收集（强制提问）

先向用户提问并拿到以下输入后，才允许写 `dev_spec.md`：
- 项目概述（必填）
- 核心特点（必填）
- 技术选型（可选，但必须确认“已定/未定”）

若缺失，必须继续追问，禁止直接写规格。  
建议首轮提供输入框：
- 让用户提供项目概述（必须）
- 让用户提供核心特点（必须）
- 让用户确认技术选型（可选，但必须确认“已定/未定”）

### Phase 2：需求确认（先确认再设计）

先输出“需求理解摘要”并请用户确认：
- 业务目标
- 范围边界（MVP/非目标）
- 关键约束（性能/成本/安全/集成边界）

未确认前，禁止进入架构设计。

### Phase 3：技术确认（状态化管理）

对技术项逐条标记：
- 已确认：记录“方案 + 理由”
- 未确认：记录“候选 + 决策条件 + 最晚决策点”

若未确认项影响主架构，必须再次询问用户后再继续。

### Phase 4：架构与模块设计

仅基于“已确认信息”设计：
- 系统边界
- 模块职责
- 数据流与依赖
- 错误处理与降级

写作中出现关键歧义时，立即暂停并向用户确认。

### Phase 5：文档生成

1. 先生成 `dev_spec.md`（严格遵循 `reference/spec-format.md` 的章节结构）
2. 再从 `dev_spec.md` 派生 `tasks.md`
3. 再从 `dev_spec.md + tasks.md` 派生 `checklist.md`

### Phase 6：回看确认与修正

输出“待确认问题清单”并请求用户确认。  
用户补充后，回填并更新三份文档。

## 不明确时的处理规范

任何不明确且影响设计质量的问题都必须触发用户确认。  
禁止默认拍板以下事项：
- 核心业务流程变化
- 技术主干切换（存储、消息、网关、部署模式）
- 安全与合规策略
- 跨系统契约假设

## 输出契约

### 1) dev_spec.md

章节结构必须与 `reference/spec-format.md` 一致，至少包含：
- 项目概述
- 核心特点
- 技术选型
- 测试方案
- 系统架构与模块设计
- 项目排期
- 可扩展性与未来展望

允许补充内容，但不能破坏上述主结构顺序。

### 2) tasks.md

必须满足：
- 从 `dev_spec.md` 映射而来
- 阶段化组织任务
- 每个任务包含：目标、输入、输出、依赖、完成定义（DoD）
- 标注并行项与阻塞项
- 为未确认事项创建 Spike/决策任务

### 3) checklist.md

必须满足：
- 使用 `- [ ]` 可勾选格式
- 覆盖需求、架构、实现、测试、发布
- 每条检查项可客观判定通过/不通过
- 包含风险复核与回滚预案检查
- 包含“待确认问题已关闭”检查

## 占位格式（未确认项）

统一写法：
- `状态：待确认`
- `候选方案：A / B / C`
- `决策依据：性能、复杂度、成本、团队熟悉度`
- `最晚决策点：里程碑 X 前`
- `需用户确认：是`

## 输出前自检

1. 是否完成 Phase 1/2/3 的用户确认
2. 是否已先提问并拿到“项目概述 + 核心特点”
3. `dev_spec.md` 是否严格遵循 `reference/spec-format.md`
4. 是否存在未确认却被当作既定事实的内容
5. 每个核心特性是否映射到任务与验收
6. 待确认问题是否可被后续关闭

任一不通过，先修正再输出。
